\section{Hiện thực ứng dụng}

Frontend \path{apps/web} cung cấp giao diện cho người dùng cuối. Các backend services Spring Boot triển khai nghiệp vụ cốt lõi như xác thực, khóa học, nội dung, tiến độ, bài kiểm tra và thử thách. Gateway dùng Spring Cloud Gateway để làm entry point, định tuyến và kiểm soát access. AI Tutor service dùng FastAPI ở mức source/prototype để phù hợp với pipeline AI/RAG; trong bản demo GKE chính, nhóm chưa đưa service này vào danh sách deployable.

Mỗi service có Dockerfile riêng, test resources và migration database riêng. Các service Java dùng Maven Wrapper để build và test nhất quán. Frontend dùng npm scripts cho lint, typecheck, test và build.

\section{Hiện thực GitHub Actions}

Bốn workflow chính đã được tạo:

\begin{itemize}
  \item \path{ci-pr.yml}: kiểm tra pull request.
  \item \path{cd-main.yml}: build, Trivy scan, tạo SBOM, push image, resolve digest, ký, attest SBOM, verify signature/attestation và mở deploy PR.
  \item \path{infra-plan.yml}: OpenTofu fmt/validate, Checkov scan và remote plan thủ công.
  \item \path{post-deploy-check.yml}: kiểm tra rollout GKE và endpoint smoke test sau deploy.
\end{itemize}

Trong CD workflow, danh sách service deployable gồm web, gateway, auth-service, course-service, content-service, progress-service, assessment-service và challenge-service. AI Tutor được ghi nhận ở source/kiến trúc nhưng không nằm trong runtime GKE chính của bản demo hiện tại.

\section{Cosign verification automation}

Trong workflow hiện tại, chuỗi sau khi push image là:

\begin{lstlisting}[language=bash]
Push -> Resolve digest -> Cosign sign
     -> Cosign attest SBOM
     -> Cosign verify
     -> Cosign verify-attestation
     -> Open deploy PR
\end{lstlisting}

Bước verify dùng certificate identity của workflow \path{.github/workflows/cd-main.yml} trên branch hiện tại và issuer \path{https://token.actions.githubusercontent.com}. Nhờ đó deploy PR chỉ được tạo sau khi chữ ký image và CycloneDX SBOM attestation được xác minh từ đúng GitHub Actions identity.

\section{Kubernetes manifests và overlay DevSecOps}

Kubernetes manifests được chia thành base và overlay. Base trong \path{infra/k8s/base} định nghĩa workload chung, gồm Deployment, Service, ServiceAccount, ConfigMap, probes, resources và securityContext. Overlay \path{infra/k8s/devsecops/apps} deploy workload vào namespace \texttt{sagelms-devsecops}, thêm ExternalSecret, Ingress, imagePullSecret và patch phù hợp với GKE.

Các platform manifests trong \path{infra/k8s/devsecops} bao gồm Harbor, FluxCD, CloudNativePG, Cloudflare Tunnel và app overlay. Cách tách này giúp báo cáo thể hiện rõ ranh giới giữa workload ứng dụng và nền tảng vận hành.

\section{Harbor registry storage trên GCS}

Repo hiện có phần hạ tầng riêng cho Harbor registry storage tại \path{infra/opentofu/modules/harbor-registry-storage}. Module này tạo GCS bucket \path{gs://sagelms-devsecops-harbor-registry}, Google Service Account \texttt{sagelms-devsecops-harbor-gcs@sagelms.iam.gserviceaccount.com}, IAM tối thiểu cho bucket và Workload Identity binding cho Kubernetes ServiceAccount \texttt{harbor/harbor-registry}. Biến \path{enable_harbor_registry_storage} trong environment \texttt{devsecops} mặc định bật để mô hình hóa phần storage này bằng OpenTofu.

Ở runtime, file \path{infra/k8s/devsecops/harbor/values-gcs.yaml} override Harbor Helm values để registry dùng \texttt{persistence.imageChartStorage.type=gcs} và bucket \texttt{sagelms-devsecops-harbor-registry}. Nếu Harbor trước đó đã chạy bằng PVC, repo cung cấp job \path{infra/k8s/devsecops/harbor/registry-pvc-to-gcs-job.yaml} để \texttt{gcloud storage rsync} dữ liệu \path{/storage} sang GCS trước khi đổi backend. Báo cáo cần phân biệt hai lớp: OpenTofu tạo bucket/IAM/Workload Identity, còn thao tác copy PVC sang GCS và helm upgrade Harbor là bước runtime migration có runbook riêng.

\section{CloudNativePG và Redis}

Database baseline của cloud runtime dùng CloudNativePG trên GKE. Cluster \texttt{sagelms-data/sagelms-postgres} có service RW \path{sagelms-postgres-rw.sagelms-data.svc.cluster.local:5432}. Backup foundation dùng bucket \path{gs://sagelms-cnpg-backup-sagelms} và object store path \path{gs://sagelms-cnpg-backup-sagelms/sagelms-postgres}. Evidence cho thấy cluster ở trạng thái healthy, WAL archive thành công và manual backup đã completed lên GCS, gồm base backup và các WAL objects.

Redis được provision bằng Memorystore với private IP, AUTH và transit encryption. Vì Redis bật TLS, các service client cần được kiểm thử cấu hình trust-store trước khi rollout chính thức.
